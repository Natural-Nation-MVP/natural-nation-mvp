# NNOS-UX-001 — Explainability First

Status: Founder Approved

## Purpose

Founder OS must keep pages simple while making every AI-generated recommendation transparent, traceable, and reviewable.

## Core Principle

The interface stays simple. The intelligence stays deep.

Founder Intelligence is a platform service, not a standalone sidebar application. It appears contextually through progressive disclosure.

## Primary UX Rule

Every recommendation generated by Founder OS must expose an **Explain** action.

The action label is:

> Explain

Do not use "Why?" as the primary label.

## Explain Drawer

Selecting **Explain** opens a contextual side drawer rather than navigating to another page.

The drawer contains three tabs:

### Explain

Shows:

- recommendation
- plain-language reasoning
- confidence
- expected impact
- estimated effort when relevant

### Sources

Shows the canonical documents and records used to generate the recommendation.

Each source must include:

- document title
- source authority level
- what the source was used for
- a direct link to open the document

### History

Shows how the recommendation changed as project conditions, approvals, or evidence changed.

## Source-Linking Rule

Reasoning must link back to the exact documents that informed it whenever those documents exist.

Source links should open through Founder OS document viewing when available. Until an internal document viewer exists, links may open the canonical GitHub source.

## Source Authority Order

Founder OS must rank evidence in this order:

1. Founder-approved decisions
2. Approved architecture and governance records
3. Approved workspace blueprints
4. Canonical repository state
5. Verified implementation and build status
6. AI analysis and recommendations
7. Brainstorming, conversations, and planning notes

Lower-authority evidence may inform a recommendation but may not override higher-authority evidence.

## Progressive Disclosure Rule

Primary screens should answer only:

1. Where am I?
2. What am I doing?
3. What should I do next?

Detailed reasoning, evidence, sources, and history remain hidden until the Founder selects **Explain**.

## Trust and Safety Rules

- Never present an AI inference as a Founder-approved decision.
- Clearly distinguish approved facts, verified state, recommendations, and ideas.
- Show uncertainty through confidence values and plain-language qualifiers.
- Do not expose hidden chain-of-thought or private internal reasoning.
- Provide concise, user-facing rationale supported by visible evidence.
- No recommendation may trigger protected or repository-changing action without Founder approval.

## Applicability

This standard applies to:

- Workspace Discovery
- Workspace Blueprint recommendations
- application recommendations
- next-action recommendations
- Build Studio recommendations
- deployment recommendations
- AI Team recommendations
- future Founder OS applications

## Acceptance Criteria

A recommendation is compliant when:

- an Explain action is available
- the page remains uncluttered before opening Explain
- reasoning is written in plain language
- confidence is visible
- supporting evidence is identified
- canonical sources are linked
- source authority is clear
- recommendation history is available when history exists

## Architectural Placement

Founder Intelligence operates as an invisible platform service shared by Founder OS applications.

It must not be added as a permanent primary-navigation page solely to display reasoning data.
